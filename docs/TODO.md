### 알아두면 좋은 것
- **TWA 앱 실행 시 뜨는 "Chrome에서 실행 중" 알림은 이 저장소 코드로 고칠 수 없는 영역(v0.3.26에서 조사·확인)** — v0.3.24에서 고친 "주소창이 뜨는" 문제(Digital Asset Links 미검증)와는 별개 현상. TWA는 Chrome 렌더링 엔진·로그인 세션·쿠키를 그대로 공유하기 때문에, 사용자가 "이게 진짜 앱인지 그냥 웹사이트인지" 헷갈리지 않도록 Android/Chrome이 앱 실행 시 이 알림을 시스템 레벨에서 직접 띄움(PWABuilder/`android-browser-helper` 공식 GitHub 이슈에도 같은 문의가 다수 있고, 웹 코드·`manifest.json`으로 끌 수 있는 설정이 아니라는 게 공식 답변으로 확인됨). 이 저장소엔 애초에 Android 네이티브 프로젝트(`AndroidManifest.xml`, `twa-manifest.json` 등)가 없고 `.aab`는 PWABuilder가 외부에서 별도로 생성하므로, 정말 이 알림 자체를 없애고 싶다면 PWABuilder 재생성 시의 옵션을 찾아보거나 TWA 대신 다른 패키징 방식을 검토해야 함(현재는 범위 밖으로 보류)
- **`createCustomToken()`을 쓰는 커스텀 인증(카카오 로그인 등)은 IAM에 `Service Account Token Creator` 역할이 미리 있어야 동작함** — 신규 Firebase 프로젝트엔 기본으로 없어서 `Permission 'iam.serviceAccounts.signBlob' denied` 에러가 남(v0.3.8에서 실전으로 확인). Google Cloud Console → IAM → `{프로젝트번호}-compute@developer.gserviceaccount.com` 계정에 그 역할을 추가해야 함 — Firestore 보안 규칙처럼 콘솔에서만 설정 가능한 영역이라 코드로는 못 고침. 한 번 해두면 카카오 외 다른 커스텀 인증(네이버 등)을 추가할 때도 이 단계는 다시 안 겪음
- **카카오 로그인 Redirect URI는 JavaScript 키·REST API 키 양쪽에 각각 등록해야 함(v0.3.7에서 실전으로 확인)** — Kakao Developers 콘솔이 키(JavaScript 키/REST API 키)마다 독립적으로 Redirect URI·도메인 화이트리스트를 관리하는 구조라, REST API 키 쪽에만 등록하면 JavaScript 키를 쓰는 흐름에서 `KOE006`(등록 안 된 리다이렉트 URI) 에러가 남 — 에러 페이지에 나오는 "사용한 리다이렉트 URI" 값이 등록한 값과 글자 그대로 똑같아 보여도 이 문제일 수 있음. 두 키 모두에 동일한 값을 등록할 것(`docs/product-specs/kakao-login.md` 참고)

- **[근본 원인 발견·수정, v0.3.23, 옹짐꾼님 확인 대기 중]** 성장 그래프(태아·출생 후 둘 다)에서 실제 기록이 있는 정상 화면인데도 "○○ 기록이 아직 없어요" 안내 문구가 옆에 겹쳐 보임(옹짐꾼님 제보, 스크린샷 포함, 2026-07-17) — v0.0.29에서 고친 "태아↔출생 후 화면 전환 시 안내문 겹침" 버그와 증상은 비슷했지만 다른 코드 경로의 문제였음. `js/growthChart.js`의 `renderFetalChart()`·`renderChart()` 둘 다 "기록 없음" 경로만 `canvas.parentElement.innerHTML`을 새로 그려 형제 요소를 정리하고, "기록 있음"(정상) 경로는 기존 캔버스에 그리기만 해서 직전에 남아있던 안내 문구가 안 지워졌던 것 — 그래프를 그리기 직전 캔버스를 새로 만들어 형제 요소를 확실히 제거하도록 수정

- **[근본 원인 발견·수정, v0.3.18, 옹짐꾼님 확인 완료 ✅]** PC 설정 탭 버튼 먹통 + 모바일 Legend 뱃지 급속 깜빡임(둘 다 옹짐꾼님 제보, 2026-07-17) — 별개 버그인 줄 알았는데 사실 하나였음. 옹짐꾼님이 보내주신 콘솔 캡처에서 Firestore `resource-exhausted: Write stream exhausted maximum allowed queued writes` 에러를 발견 → `js/push.js`의 `refreshTokenIfNeeded()`가 앱이 데이터를 불러올 때마다 FCM 토큰을 조건 없이 재저장(매번 `updatedAt`이 바뀌어 "문서 변경"으로 처리됨)하면서 이 기기 자신의 `onSnapshot`을 다시 울리고, 그게 또 저장을 부르는 **무한 저장 루프**였음(`debounceSave()`를 안 거치는 직접 저장이라 v0.3.16 가드로도 못 막았음). 이 루프가 도는 동안 현재 화면이 계속 통째로 다시 그려져서 PC에서는 클릭이 씹히고(시간이 지나 Firestore 백오프가 걸리면 느려지며 클릭이 먹힘) 모바일에서는 뱃지가 깜빡였던 것 — 이 루프는 푸시 알림 권한이 켜진 기기에서만 돌아서 PC엔 없고 모바일에만 있었던 것도 설명됨. **가족 그룹 공유 계정은 완전히 우연히 이 버그를 피해감** — `subscribeToUserData()`가 `familyId`가 있으면 `users/{uid}` 변경으로 `onData()`를 다시 안 부르도록 짜여 있어서(실제 데이터는 `families/{familyId}`에서만 옴), 이 루프가 반복 저장한 `users/{uid}`의 변화 자체를 감지하지 않았기 때문. 토큰이 실제로 바뀌었거나 24시간 지났을 때만 저장하도록 수정해서 해결(자세한 내용은 CHANGELOG.md v0.3.18 참고) — 옹짐꾼님이 "드디어 해결됐어"로 확인해주심(2026-07-17)
- **[v0.3.21까지 총 4단계에 걸쳐 수정, 옹짐꾼님 확인 대기 중]** 모바일 알림 권한/화면 관련 버그(옹짐꾼님 제보, 2026-07-17) — 증상이 계속 모습을 바꿔가며 나타나서 여러 버전에 걸쳐 수정함: (1) v0.3.14/15: 팝업 최대 3번→2번, (2) v0.3.19: `waitForGrantedPermission()` 대기를 250ms→2.5초로 늘려 **팝업 중복 해결(옹짐꾼님 확인)**, (3) v0.3.20: 그 대신 "허용해도 화면이 안 바뀐다"는 증상 발견 → `result` 값을 직접 화면에 전달하도록 수정, (4) v0.3.21: "2.5초 대기가 체감상 길다" + "하위 설정 바꾸거나 끄기 누르면 매번 알림 받기로 돌아간다"는 재제보 → **진짜 원인은 그 모바일 브라우저가 한 번 명백히 허용된 뒤에도 `Notification.permission` 라이브 값을 계속 잘못(`'default'`) 보고하는 것**(권한 자체는 정상 저장되어 있음, 브라우저가 JS에 보고를 잘못 하는 것뿐)이었음 → 라이브 값 대신 "한 번이라도 확실히 허용된 걸 확인했다"는 로컬 기억을 우선 신뢰하는 `effectivePermission()`을 도입해서 모든 렌더/체크 경로를 교체하고, 대기는 화면과 분리해 체감 지연도 없앰 — 재현되면 알려주세요(위 "현재 확인 필요 항목" 참고)
