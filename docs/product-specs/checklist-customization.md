# 체크리스트 커스터마이징 (v0.0.40)

> 상태: ✅ 코드 완료, 실기기 확인 대기 (별도 배포 절차 없음 — 다른 앱 본체 기능과 함께 Vercel 자동 배포)

## 배경

기존 체크리스트(예방접종/발달/이유식/정부지원)는 전부 **월령·주차로 인덱싱된** 구조였음. 앞으로 옹짐꾼님이 계속 추가할 예정인 카테고리(외출 준비물, 100일 준비, 돌 준비, 돌사진 준비 등)는 월령 인덱싱이 없는 **"한 번에 쭉 확인하는" 플랫 리스트**라서, 이 둘을 같은 "탭" 개념으로 묶되 렌더링은 통일했음. 탭이 계속 늘어날 걸 감안해서 "원하는 탭만 보기" · "캘린더 연동 여부 선택" · "직접 만들기"를 이번에 함께 넣음.

## 탭 종류

| kind | 예시 | 구조 |
|---|---|---|
| `indexed` | 임신 체크·예방접종·발달·이유식 | 월령/주차별 하위 카테고리 여러 개 (기존 구조 그대로) |
| `gov` | 정부지원 | `js/govSupport.js`가 별도 렌더링 (체크리스트 아님, 대상 제도 목록). v0.2.4부터 `S.customGovItems`로 사용자가 지자체별 지원금 등을 직접 추가할 수 있음 — 아래 "정부지원 직접 추가" 참고 |
| `flat` | 준비물 팩(외출 준비물 등) · 사용자 정의 체크리스트 | 하위 카테고리 없이 카테고리 1개짜리 — `getCats()`가 `[{key, label, items}]` 형태로 감싸서 반환하면 기존 `calcScore`/`getCatItems`/`renderClMain`이 그대로 처리함(신규 렌더 코드 없음) |

## 핵심 설계 — "키 기반" 탭 판별

기존엔 `S.clTab`(숫자 인덱스)이 "0=예방접종, 1=발달…"처럼 **위치가 고정**돼 있다고 가정하고 `renderContextBanner`/`autoSelectCat`/`getCats`/`getCustomKey`가 전부 숫자로 분기했음. 탭을 숨길 수 있게 되면 위치가 사용자마다/순간마다 달라지므로, 이제 이 함수들은 전부 아래 방식으로 판별함:

```js
const tabDefs = getVisibleTabDefs(child); // 지금 보이는 탭 목록
const curKey  = tabDefs[S.clTab || 0]?.key; // 'vax' | 'dev' | 'food' | 'gov_preg' | 'gov_born' | 'preg' | pack/custom key
```

`S.clTab`은 Firestore에 저장되지 않는 세션 전용 값(`js/state.js`의 `emptyState()`에서 초기화만 되고 `saveState()`엔 포함 안 됨)이라, 이 의미 변경이 기존 저장 데이터에 영향을 주지 않음.

## 데이터 모델 (Firestore, additive)

```json
{
  "customChecklists": [
    { "id": 1720000000000, "key": "custom_1720000000000", "label": "어린이집 준비물", "icon": "checklist", "stage": "born",
      "items": [{ "id": "cclt_1720000000000_0", "t": "가방", "r": true }] }
  ],
  "clSettings": {
    "hiddenTabs": ["food"],
    "calendarSync": { "vax": false }
  }
}
```

- `customChecklists`: 사용자가 "새 체크리스트 만들기"로 만든 플랫 체크리스트. 항목 id는 `cclt_`로 시작(기존 "＋ 항목 직접 추가하기"가 만드는 `custom_` 접두어와 겹치면 삭제 버튼이 엉뚱한 배열을 건드리게 됨 — 반드시 접두어를 분리할 것). `stage`(`'preg'`|`'born'`, v0.0.41 추가)가 없으면 `'born'`으로 취급(v0.0.40에서 만든 기존 데이터 하위 호환)
- `clSettings.hiddenTabs`: 숨긴 탭의 key 배열(내장 6종 + 팩 13종 + 커스텀 공용). 전부 숨기면 `getVisibleTabDefs()`가 안전장치로 전체를 다시 보여줌
- `clSettings.calendarSync`: 명시적으로 끈 것만 저장(기본은 켜짐) — `{ vax: false }`처럼 값이 있을 때만 꺼진 것으로 취급

## 준비물 팩 — `data/checklist-packs.js`

내장 13종(외출 준비물/100일 준비/돌 준비/돌사진 준비/태명 정하기/태교여행 준비/출산가방/산후조리원/신생아 맞이 준비/여행(외박) 준비물/어린이집(유치원) 준비/상비약 체크리스트/이유식 준비, v0.3.1에서 7종 추가)이 들어있는 배열. 새 팩을 추가하려면 이 배열에 `{ key, label, icon, stage, items }` 객체 하나만 추가하면 됨 — 렌더링 코드는 건드릴 필요 없음(`js/checklist.js`의 `packTabDefs()`가 자동으로 탭 목록에 포함시킴). `stage` 필드(`'preg'` | `'born'`, 기본값 `'born'`)로 아이 단계에 맞는 팩만 노출됨. `dd`(상세 설명) 필드는 앱의 "자세히 보기" 아코디언과 `scripts/build-guide.mjs`가 생성하는 guide 페이지 양쪽에 그대로 반영됨(v0.0.41).

## 캘린더 연동 켜기/끄기

`js/checklistCalendarLink.js`가 실제로 캘린더와 항목을 연결하는 곳은 **오직 예방접종(vax)·발달(dev) 탭**뿐임(`data/checklist-links.js`에 명시적으로 매핑된 항목만). 이유식·정부지원·준비물 팩·커스텀 체크리스트는 애초에 이 매핑 파일에 항목이 없어서 캘린더에 아무것도 만들지 않음 — 그래서 설정 화면에도 이 두 탭에만 "캘린더 연동" 토글을 보여준다(있지도 않은 연동을 끄는 척하는 UI를 만들지 않기 위한 의도적 선택).

**v0.0.42부터 토글의 의미가 바뀜** — 예전엔 완료 체크의 양방향 동기화만 멈추는 정도였는데, 이제는:
- **끄면**: `js/calendar.js`의 `getAutoEvs()`가 해당 카테고리 일정(vax → `type==='vax'`, dev → born 단계의 `type==='req'|'rec'`)을 캘린더에서 통째로 필터링해서 안 보이게 함
- **켜면**: `js/checklistCalendarLink.js`의 `resyncTabForAllChildren(tabKey)`가 모든 아이의 지금 체크리스트 상태를 기준으로 캘린더 완료 표시를 한 번에 재동기화한 뒤 다시 보여줌 — 꺼져있던 동안 체크리스트에서 바뀐 내용이 캘린더에 반영 안 돼 있던 문제를 해결

이 구조(카테고리별로 캘린더 표시를 껐다 켰다 하는 것)는 앞으로 원더윅스 기간·발달 현황 기간 표시 같은 기능에도 그대로 재사용할 수 있게 설계함(`docs/TODO.md` "다음 후보" 참고).

- **체크리스트 탭 바 오른쪽 "편집" 버튼(v0.0.43부터 아이콘+텍스트)** — 설정 탭으로 바로 이동하는 바로가기(중복 UI를 만들지 않기 위해 실제 관리 기능은 설정 탭에만 있음). 바로 아래에 짧은 안내문(`#clTabHint`)이 있어서 처음 보는 사람도 이 버튼이 뭘 하는지 알 수 있음(예전엔 이 안내문이 설정 탭에 있었는데, 옹짐꾼님 피드백으로 실제 버튼이 있는 체크리스트 탭 쪽으로 옮김)
- **커스텀 체크리스트 편집(v0.0.41)** — 설정 탭의 각 커스텀 체크리스트 행 옆 연필 아이콘. 이름·항목(텍스트/필수여부/추가/삭제)을 한 화면에서 통째로 고칠 수 있음. 기존 항목은 id를 그대로 유지해서 그동안 체크한 기록이 끊기지 않고, 새로 추가한 행만 새 id(`cclt_{체크리스트id}_{timestamp}_{순번}`)를 받음

## 준비물 팩 vs 커스텀 체크리스트 — 항목 저장 방식이 다름 (v0.0.44 정리)

체크리스트 탭에서 "＋ 항목 직접 추가하기"를 눌렀을 때 어디에 저장되는지가 카테고리 종류에 따라 다르다 — 이 차이를 헷갈리면 "설정에서 편집한 게 반영이 안 된다" 같은 버그를 만들기 쉬우니 주의할 것:

| | 준비물 팩(`pack_*`) | 커스텀 체크리스트(`custom_*`) |
|---|---|---|
| 기본 항목 출처 | `data/checklist-packs.js`(콘텐츠, `dd` 상세 설명 포함) — 수정 불가 | 사용자가 처음 만들 때 직접 입력 — 전부 수정 가능 |
| "+ 항목 추가"로 만든 항목 저장 위치 | `S.customClItems[{child.id}_{packKey}]` (**아이별로 따로**) | `S.customChecklists[i].items` (**모든 아이가 공유하는 템플릿에 바로**) |
| "내가 추가함" 배지 | 있음(기본 콘텐츠와 구분하기 위해 의미 있음) | 없음(v0.0.44부터 — 전부 본인이 만든 목록이라 구분 무의미) |
| 삭제 가능 범위 | 직접 추가한 것만 | 전부 |
| 설정 탭 "편집"이 읽는 곳 | 지금 선택 중인 아이(`S.selC`)의 `customClItems` 버킷만(`openEditPackExtrasModal`) | `cl.items`(+ 아직 남아있는 레거시 `customClItems`가 있으면 함께 병합해서 보여주고, 저장 시 `cl.items`로 완전히 옮김) |

**왜 이렇게 다른가**: 준비물 팩은 여러 아이가 같은 콘텐츠(기본 항목)를 보되, "우리 아이만의 추가 준비물"은 아이마다 다를 수 있어서 아이별로 분리해둔 것. 커스텀 체크리스트는 애초에 콘텐츠랄 게 없고(전부 사용자 입력) 아이 구분 없이 하나의 목록을 계속 고쳐 쓰는 쪽이 자연스러워서 처음부터 공유 템플릿 하나로 관리한다.

**v0.0.43의 버그**: 커스텀 체크리스트도 준비물 팩과 똑같이 "+ 항목 추가"가 아이별 `customClItems`에 쓰도록 돼있었는데, 설정 탭 "편집"은 `cl.items`만 읽고 썼음 — 그래서 체크리스트 탭에서 추가한 건 설정에 안 보이고, 설정에서 수정한 것만 반영되는 반쪽짜리 동기화였음. v0.0.44에서 커스텀 체크리스트의 "+ 항목 추가"가 `cl.items`에 바로 쓰도록 고쳐서 완전한 양방향 동기화가 됨(`js/checklist.js`의 `submitAddClItem()`/`deleteCustomChecklistItemInTab()` 참고). 이미 아이별 버킷에 남아있던 레거시 항목은 설정에서 편집을 한 번 열고 저장하면 자동으로 `cl.items`에 합쳐짐.

**범위 제한(준비물 팩 한정)**: 예방접종·발달·이유식처럼 월령/단계별로 여러 하위 카테고리가 있는 탭은 카테고리마다 별도 저장 키(`{child.id}_{catKey}` 또는 `__vax`/`__dev` 접미사)를 쓰기 때문에, 탭 하나에 "편집" 버튼 하나로 매핑되지 않아 설정 탭 편집 대상에서 제외함. 확장하려면 먼저 "어느 카테고리(월령)의 항목을 볼지" 고르는 UI가 필요함(커스텀 체크리스트는 애초에 하위 카테고리가 없어서 이 문제가 없음).

**v0.3.1: 예전 예외였던 "출산 준비물"(prep) 폐지** — v0.2.4에서는 `kind: 'indexed'`로 분류돼 있지만 실제로는 `clData.preg`의 `preg_prep` 카테고리 단 하나뿐이라 위 범위 제한이 적용되지 않는 예외로 두고, 설정 화면 row key(`prep`)를 `BUILTIN_CAT_KEY_MAP`으로 `preg_prep`에 매핑해서 편집을 지원했었음. v0.3.1에서 이 단일 탭 자체를 없애고 "출산가방"·"산후조리원"·"신생아 맞이 준비" 등 여러 준비물 팩(`stage: 'preg'`)으로 세분화하면서, 이 예외와 `BUILTIN_CAT_KEY_MAP`의 `prep` 항목도 함께 제거함(옹짐꾼님 요청 — "출산 준비물이 너무 포괄적") — 지금은 `BUILTIN_CAT_KEY_MAP`이 빈 객체이고, 새 팩들은 처음부터 팩과 동일한 방식(`pack_*` key, tab key = cat.key)으로 만들어져서 이런 매핑이 필요 없음.

## 정부지원 직접 추가 (v0.2.4)

`data/government-support.js`의 전국 단위 일정 외에, 지자체마다 다른 지원금처럼 앱에 없는 항목을 사용자가 직접 추가하는 기능. 커스텀 체크리스트처럼 아이별이 아니라 **가족 전체가 공유하는 목록**(`S.customGovItems`)이고, `stage`(`'preg'`|`'born'`) 필드로 설정 화면에서 임산부용/육아용을 구분해서 관리함(실제 노출은 지금 아이의 `stage`와 일치하는 항목만).

- **입력 방식**: 제목·날짜(직접 선택, 임신 주차나 출생 후 개월수 같은 상대 계산 아님)·중요도(필수/해당자)·설명(선택)·신청 링크(선택). 날짜를 상대 계산이 아니라 절대값으로 받는 이유는 지자체 지원금처럼 대부분 "이 시점까지 신청"이 이미 정해져 있는 일회성 정보라 관리가 단순한 쪽을 택함(옹짐꾼님 확인 후 결정, 2026-07-15) — 여러 아이에게 자동으로 재계산해서 적용하고 싶다면 나중에 상대 시점 입력 방식을 별도로 추가할 수 있음
- **진입점**: 설정 → 체크리스트 관리(임산부용/육아용 그룹 각각의 "정부지원" 행 옆 편집 연필, v0.2.5부터 "공통" 그룹 없이 이 두 자리로 통합됨), 또는 체크리스트 탭 → 정부지원 탭 하단 "＋ 항목 직접 추가하기" 버튼 — 둘 다 같은 모달(`openGovItemsModal(stage)`)을 열며 완전히 동일하게 동작함
- **렌더링**: `js/calendar.js`의 `getAutoEvs()`가 `S.customGovItems`를 읽어서 기존 govSupportSchedule 항목과 동일한 모양(`type:'gov'`)으로 섞어 넣음 — 그래서 별도 렌더 코드 없이 캘린더·체크리스트 정부지원 탭에 기존 항목과 완전히 같은 방식(상태 변경·마감 임박 표시 등)으로 표시됨. `customGov: true`/`_customGovId` 필드로 기존 항목과 구분해서 삭제 버튼을 추가로 보여줌
- **삭제**: 정부지원 탭 목록의 항목별 ✕ 버튼, 또는 설정의 관리 모달에서. 삭제 시 그동안 저장된 신청 상태(`S.eventMods`)도 함께 정리함. **v0.2.5 버그 수정**: `deleteCustomGovItem()`이 호출된 곳과 상관없이 무조건 관리 모달(`openGovItemsModal`)을 다시 열어버려서, 체크리스트 탭 목록의 ✕(모달이 안 열려있는 상태)에서 삭제해도 갑자기 "항목 추가" 모달이 뜨는 버그가 있었음 — 이제 관리 모달이 실제로 열려있을 때(`#govItemsList`가 DOM에 있을 때)만 다시 열도록 수정
- **보안**: 제목·설명·링크 전부 사용자 입력이라, 렌더링하는 모든 자리(체크리스트 탭·캘린더 월/주간 그리드·날짜 상세 패널·일정 수정 모달)에서 `escapeHtml()`을 거침. 링크는 추가로 `sanitizeUrl()`을 거쳐 `http`/`https`가 아닌 스킴(`javascript:` 등)은 무조건 빈 문자열로 걸러냄 — 이 작업 중에 캘린더 월/주간 그리드(`renderEventLine`/`renderWeekTimedBlock`)가 이벤트 제목을 이스케이프 없이 그리고 있던 걸 발견해서 함께 고침(기존 "내 일정"(custom) 이벤트에도 있던 문제였음, `docs/CHANGELOG.md` v0.2.4 참고)

## 알려진 한계 (다음 후보)

- 탭 순서를 사용자가 직접 바꾸는 기능(드래그 정렬)은 없음 — 항상 내장 → 팩 → 커스텀 순서로 고정
- 준비물 팩·커스텀 체크리스트를 캘린더에 표시하는 기능은 없음(원래 목적이 "캘린더를 지저분하게 하지 않는 것"이라 의도적으로 없음) — 필요해지면 별도 논의 필요
- guide 페이지에 반영된 준비물 팩 콘텐츠는 앱 체크리스트와 `dd` 필드를 그대로 공유하므로, 내용을 고치려면 `data/checklist-packs.js` 한 곳만 수정하고 `node scripts/build-guide.mjs`를 재실행하면 됨(둘이 따로 관리되지 않음)
