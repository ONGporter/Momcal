## 큰 그림

맘캘은 이미 PWA(설치 가능한 웹앱)라서, 코드를 새로 짜지 않고 **TWA(Trusted Web Activity)** 로 감싸기만 하면 플레이스토어에 낼 수 있어요. TWA는 얇은 Android 앱 껍데기이고, 실제 화면은 그대로 momcal.app 웹사이트를 보여줘요 — 주소창만 없어서 네이티브 앱처럼 보여요. **웹사이트를 업데이트하면 앱스토어 앱도 따로 재배포할 필요 없이 자동으로 최신 내용이 뜨는 게 제일 큰 장점**(단, 스토어에 새로 올릴 필요가 있는 건 아이콘·버전명처럼 AAB 안에 박제된 것들뿐).

**전체 순서**(중요 — 아래에서 하나씩 설명):
1. PWA 상태 점검 (사실상 이미 완료)
2. TWA로 패키징 → 이 과정에서 패키지 이름 + 서명 키가 생김
3. 서명 키 백업 (가장 중요, 잃어버리면 되돌릴 수 없음)
4. Digital Asset Links 설정 (주소창 없애기)
5. Google Play 개발자 계정 만들기
6. **Android 개발자 인증** — 2026년 4월에 새로 생긴 단계, 앱 만들기 *전에* 완료해야 함
7. 앱 만들기 + 스토어 등록정보(설명·스크린샷 등)
8. 콘텐츠 등급 / 타겟층 / 데이터 보안 설문
9. 광고 선언
10. 비공개 테스트(테스터 12명·14일) → 프로덕션 신청 → 심사

가장 오래 걸리는 두 가지가 **"기다려야 하는" 종류**예요. 그래서 이 둘을 제일 먼저 시작하고, 기다리는 동안 나머지(스토어 문구·스크린샷 등)를 준비하는 게 시간을 아끼는 방법이에요.

1. **테스터 12명 모으기 시작** — 비공개 테스트는 테스터 12명이 **14일 연속** 참여해야 통과돼요. 지인·커뮤니티에서 미리 섭외해두세요(Google 계정 필요, 초대 링크로 참여).

## 1단계 — PWA 상태 점검 (거의 다 돼있음)

`manifest.json`을 확인해보니 TWA 패키징에 필요한 게 이미 다 있어요:

- ✅ `name`/`short_name` 있음
- ✅ `icons`에 192·512 사이즈 + **maskable 버전**까지 있음(Android 적응형 아이콘에 필수)
- ✅ `display: standalone` (TWA 요건 충족)
- ✅ `start_url`/`scope`, `theme_color`/`background_color` 있음(스플래시 화면에 쓰임)
- ✅ `icons/icon-512.png`가 이미 알파 채널 없는 불투명 PNG — 플레이스토어 앱 아이콘 규격(512×512, 32비트, **투명 배경 불가**)에 바로 쓸 수 있음

추가로 확인해두면 좋은 것(필수는 아님):
- [x] `manifest.json`에 `"id": "./"` 필드 추가(v0.3.3에서 완료) — PWABuilder 분석에서 경고로 나왔던 것
- [x] `manifest.json`에 `categories` 필드 추가(v0.3.3에서 완료, `["health", "lifestyle", "parenting"]`)
- [ ] Chrome DevTools → Lighthouse → PWA 카테고리로 마지막 점검 (선택, 안 해도 무방)

> **PWABuilder 분석 결과 메모(2026-07-16)**: momcal.app을 PWABuilder로 분석했을 때 오류(Error) 0건, 경고(Warning) 3건(서비스워커 미검출·스크린샷 없음·id 없음), 정보(Info) 6건이 나왔음. "서비스워커 미검출"은 `js/app.js`가 `window.addEventListener('load', ...)` 안에서 등록하는 타이밍을 크롤러가 못 잡은 **오탐**으로 확인됨(실제로는 정상 등록·동작 중, 코드 수정 안 함). "스크린샷 없음"은 Chrome 설치 UI를 풍성하게 해주는 선택 항목이라 스킵(플레이스토어 등록정보용 스크린샷은 7단계에서 별도 준비). "id 없음"은 위에서 해결함.

## 2단계 — TWA로 패키징

두 가지 도구가 있어요:

| 도구 | 특징 |
|---|---|
| **PWABuilder** (https://www.pwabuilder.com) | 웹사이트에 URL만 입력하면 매니페스트를 자동 분석해서 Android 패키지(.aab)를 바로 다운로드해줌. Android Studio 없이도 됨. 제일 빠르게 감 잡기 좋음 |
| **Bubblewrap CLI** (Google 공식, `@bubblewrap/cli`) | Node.js 기반이라 지금 이 프로젝트 환경과 잘 맞음. `npm install -g @bubblewrap/cli` → `bubblewrap init --manifest=https://momcal.app/manifest.json` → 질문에 답하면 Android 프로젝트 생성 → `bubblewrap build`로 .aab 빌드. 나중에 앱 업데이트할 때도 CLI로 재빌드하기 편함 |

**추천**: 처음엔 PWABuilder로 빠르게 한 번 만들어서 실제 폰에 설치해보고 감을 잡은 뒤, 이후 업데이트 자동화가 필요해지면 Bubblewrap으로 넘어가세요.

이 단계에서 정해야 할 것:
- **패키지 이름**(예: `app.momcal.www` 또는 `com.momcal.app`) — 나중에 절대 바꿀 수 없으니 신중하게. 보통 도메인을 거꾸로 씀
- **앱 버전 코드/이름** — 웹앱의 `v0.3.2`와는 별개로, Android 쪽 버전 코드를 정수로 관리해야 함(업데이트할 때마다 +1)

이 과정에서 **서명 키(keystore)** 와 **SHA-256 인증서 지문**이 생성돼요 — 다음 단계에서 이걸 씁니다.

> ✅ **완료(2026-07-16)**: PWABuilder로 생성함. Package ID `app.momcal.www`, App version `1.0.0.0`, Signing key는 New 선택(PWABuilder가 비밀번호 자동 생성 → `signing-key-info.txt`에 기록됨). 결과물(.aab/.apk/keystore/signing-key-info.txt/assetlinks.json) 확인 완료 — .aab는 정상 AAB(JAR) 포맷, .apk는 서명 블록 포함해서 정상 서명 확인됨.

## 3단계 — 서명 키 백업 (제일 중요 — 되돌릴 수 없음)

Bubblewrap/PWABuilder가 만든 keystore 파일(`.jks`/`.keystore`)과 비밀번호를 **잃어버리면 이 앱을 다시는 업데이트할 방법이 없어요.** 새 앱으로 처음부터 다시 등록해야 하고, 리뷰·설치 수가 전부 날아가요.

- [x] keystore 파일 + 비밀번호를 **최소 2곳**(예: 1Password + 클라우드 백업)에 보관 — ✅ 2026-07-16 옹짐꾼님 확인 완료
- [x] Play Console의 **"Play App Signing"** 에 업로드 키로 등록해두는 걸 추천(기본으로 켜짐) — Google이 최종 서명 키를 별도로 관리해줘서, 업로드 키가 유출돼도 복구 절차가 있음
  - ✅ **완료(2026-07-18)**: 비공개 테스트로 실제 설치해보니 TWA에 주소창이 뜨는 문제 발견(옹짐꾼님 제보, 스크린샷) → Play App Signing이 기본 활성화라 실제 배포되는 앱은 업로드 키가 아닌 **Play 앱 서명 키**로 재서명됨을 확인. Play Console → "Google Play로 보호됨" → "Play 스토어 보호" 에서 앱 서명 키 인증서 SHA-256 지문(`1A:02:2D:...5D:E3`) 확인 후 `.well-known/assetlinks.json`의 `sha256_cert_fingerprints`에 업로드 키 지문과 함께 추가(배열에 둘 다 포함)

## 4단계 — Digital Asset Links (주소창 없애기)

TWA가 브라우저처럼 주소창 달고 뜨지 않고 "진짜 앱처럼" 보이려면, 도메인이 이 앱 서명과 연결됐다는 걸 증명해야 해요.

- `https://momcal.app/.well-known/assetlinks.json` 파일을 배포해야 함 — `ads.txt`처럼 루트에 정적 파일로 추가하면 됨(Vercel에서 이미 `ads.txt`가 잘 서빙되고 있으니 문제없이 될 것)
- 내용 예시(2단계에서 얻은 SHA-256 지문을 넣음):
```json
[{
  "relation": ["delegate_permission/common.handle_all_urls"],
  "target": {
    "namespace": "android_app",
    "package_name": "app.momcal.www",
    "sha256_cert_fingerprints": ["SHA256_지문_여기에"]
  }
}]
```
- 이 파일 없어도 앱은 작동하지만 브라우저처럼 주소창이 붙어서 보임 — 필수는 아니어도 강력 권장

> ✅ **완료·검증됨(2026-07-16)**: PWABuilder가 SHA-256 지문까지 채워서 생성해준 `assetlinks.json`을 `.well-known/`에 그대로 추가함(v0.3.4). Vercel 배포 후 `https://momcal.app/.well-known/assetlinks.json`에서 정상 노출 확인 완료(리다이렉트·404 없음, package_name·SHA-256 지문 정확히 일치) — 4단계 끝.

## 5단계 — Google Play 개발자 계정 만들기

- console.play.google.com 가입, **등록비 미화 25달러(1회 결제, 평생 이용)**
- 유효한 국가발급 신분증으로 본인인증 필요 — 처리에 며칠 걸릴 수 있음(위 "지금 뭐부터?" 참고)
- 개인 계정 vs 조직 계정: **개인 계정**이 간단(신분증만 필요). 조직 계정은 사업자 + D-U-N-S 번호(9자리, 발급에 며칠~몇 주)가 필요한 대신 아래 "12명 14일" 테스트 요건이 면제됨 — 사업자가 없으면 그냥 개인 계정으로 진행하는 게 맞음

> ✅ **완료(2026-07-16)**: 옹짐꾼님 개발자 계정 신원 확인 통과 — 5단계 끝. 2단계에서 이미 SHA-256 지문(`.well-known/assetlinks.json`에 반영된 값과 동일)을 확보해둔 상태라 바로 6단계 진행 가능.

## 6단계 — Android 개발자 인증 ⚠️ (2026년 4월 신규 정책, 주의)

**이 부분은 최근에 막 생긴 단계라 꼭 짚고 넘어갈게요.** 2026년 4월부터 Google Play Console에 새 앱을 등록하기 *전에*, "Android 개발자 인증"이라는 별도 절차로 **패키지 이름 + 서명 키의 SHA-256 인증서 지문을 먼저 등록하고 Google 승인을 받아야** 앱 만들기를 시작할 수 있어요(기존엔 앱부터 만들고 나중에 서명했는데, 순서가 바뀜).

그래서 **2단계(TWA 패키징)를 먼저 끝내서 SHA-256 지문을 손에 쥐고 있어야** 이 단계를 진행할 수 있어요.

- [ ] Play Console에서 새 앱 만들기를 시작하면 이 인증 화면이 먼저 뜸 — 패키지 이름과 SHA-256 지문을 입력
- [ ] 새로 만드는 패키지 이름이라면(기존에 아무도 안 쓴 이름) 소유권 증명 절차가 비교적 간단함(기존 이름을 가져오는 경우만 서명된 APK 업로드로 소유권 증명이 추가로 필요)
- [ ] 승인 대기 후 상태가 "등록됨"으로 바뀌면 그때부터 정상적으로 앱 만들기 진행 가능

이 정책은 도입된 지 얼마 안 돼서 세부 화면이 계속 바뀔 수 있어요 — Play Console에 실제로 들어가면 화면에서 단계별로 안내해주니 그대로 따라가면 되고, 막히는 부분이 있으면 그때 다시 같이 확인해요.

> ✅ **완료(2026-07-16)**: Android 개발자 인증 통과, Play Console에 앱 생성 완료(대시보드 진입 확인) — 6단계 끝.

## 8단계 — 콘텐츠 등급 / 타겟층 / 데이터 보안 설문

세 가지 설문을 Play Console 안내에 따라 작성해요. 아래는 참고용 방향성이고, **애매한 문항은 Play Console이 그때그때 도움말을 보여주니 그걸 기준으로 판단하는 게 제일 정확해요**(법적 판단이 필요한 영역이라 제가 대신 확답을 드리긴 어려워요).

- **콘텐츠 등급 설문**: 폭력적·선정적 콘텐츠가 없는 육아 정보 앱이라 낮은 등급이 나올 것으로 예상
- **타겟층**: 주 이용자는 부모(성인)이므로 "아동 대상 앱" 아님으로 답변하는 게 맞아 보임. 다만 **아이 이름·생년월일 등 자녀 정보를 다루는 앱**이라, "아동 관련 데이터 처리" 관련 문항은 신중하게 — 애매하면 Google Play 정책센터의 가족 정책(Families Policy) 문서를 직접 한 번 확인해보길 권해요
- **데이터 보안(Data Safety) 섹션**: 여기가 제일 꼼꼼해야 함 — 실제로 수집하는 데이터를 정확히 신고해야 하고, 허위 기재는 반려·제재 사유가 돼요. 맘캘이 Firebase로 수집하는 것 기준으로 미리 정리하면:
  - 이메일 주소(Firebase Auth)
  - 이름(사용자·자녀)
  - 생년월일(사용자·자녀 — "건강 및 피트니스" 카테고리로 분류될 가능성 있음, 임신·예방접종 일정이 여기 해당할 수 있음)
  - 앱 활동(체크리스트 완료 여부 등)
  - 기기 정보(FCM 푸시 토큰)

## 9단계 — 광고 선언 (AdSense 관련, 확인 필요)

- 앱에 광고 있음 → "예"로 표시(자동 광고 스크립트가 `index.html`에 있으므로)
- **AdSense 심사가 아직 안 끝났어도 "예"가 맞음** — Play Console 선언 기준은 "지금 실제로 광고가 보이는지"가 아니라 "앱에 광고 코드가 포함돼 있는지"라, Google이 앱을 스캔해서 광고 스크립트를 자동 감지하면 선언과 무관하게 "광고 포함" 라벨을 붙일 수 있음(공식 도움말: "앱에 광고가 포함되어 있음을 정확하게 언급해야 할 책임은 개발자에게 있지만, Google에서 언제든지 이를 확인하고 필요한 경우 '광고 포함' 라벨을 표시할 수 있음"). TWA는 momcal.app을 그대로 보여주는 구조라 AdSense 스크립트가 이미 앱 안에도 들어있는 셈 — "아니요"로 냈다가 승인 후 광고가 뜨기 시작하면 그 시점부터 선언이 부정확해지는 문제도 있음. 이 선언은 재출시 없이 Play Console에서 언제든 수정 가능하니 지금은 "예"로 두고 진행
- 다만 **AdSense(웹 광고 스크립트)가 TWA 안에서도 그대로 노출되는지, Play 정책상 별도 제약이 있는지는 실제 빌드해서 확인해보는 걸 추천**해요 — TWA는 결국 Chrome이 웹페이지를 그대로 보여주는 거라 대부분 문제없이 작동하지만, 광고 관련 정책은 Google이 종종 세부 조항을 바꾸는 영역이라 심사 전에 실기기로 한 번 확인해보는 게 안전해요

## 10단계 — 비공개 테스트 → 프로덕션 출시

**개인 개발자 계정(2023년 11월 13일 이후 생성)은 아래 요건을 반드시 통과해야 프로덕션에 낼 수 있어요:**

- [ ] 비공개 테스트(Closed Testing) 트랙 만들고 AAB 업로드
- [ ] 테스터 **12명 이상**이 참여 링크로 초대 수락 + 앱 설치(Google 계정 필요)
- [ ] 그 12명이 **최근 14일 동안 연속으로** 테스트 참여 상태 유지(중간에 이탈해서 인원이 줄면 다시 채워야 함)
- [ ] 14일 경과 후 "프로덕션 액세스 신청" 버튼이 활성화됨 → 신청 시 몇 가지 질문에 답변
- [ ] Google 검토(보통 며칠) → 승인되면 프로덕션에 동일한 AAB를 다시 업로드해서 정식 제출
- [ ] 프로덕션 심사(보통 7일 이내, 첫 제출은 더 걸릴 수 있음) → 통과하면 스토어에 게시됨

전체적으로 **테스트 시작부터 정식 출시까지 짧으면 3주, 길면 한 달 정도** 잡는 게 현실적이에요(테스터 모집이 매끄럽지 않으면 더 걸릴 수 있음). 심사 반려 시 사유가 이메일로 오고, 수정 후 재제출하면 돼요(비공개 테스트가 이미 통과된 상태라면 재제출은 프로덕션 심사만 다시 받으면 됨).
